# ⚔ D&D Portal

Веб-портал для игры в Dungeons & Dragons (Otel Ural Team-версия) с друзьями: аккаунты, коды-подписки, игровые
комнаты, загрузка карты, перетаскивание фишек персонажей в реальном времени, броски костей
с общим журналом, панель мастера и админ-панель.

**Стек:** React + Vite + TypeScript (фронтенд) · FastAPI + SQLAlchemy (бэкенд) · PostgreSQL · WebSocket.

**Архитектура:** контейнеризированный модульный монолит — 3 сервиса в Docker
(`frontend` nginx · `backend` FastAPI · `db` PostgreSQL). Это НЕ микросервисы (подробнее — в конце файла).

## 🚀 Быстрый старт: всё в Docker (рекомендуется)

Нужен только **Docker Desktop**. Из корня проекта:

```bash
copy .env.example .env       # Windows.  В bash: cp .env.example .env
# открой .env и подставь свои значения (там написано, что куда)
docker compose up -d --build
```

Файл `.env` хранит секреты (`JWT_SECRET`, пароль базы, пароль админа) и в git не попадает.
Без него `docker compose` специально не запустится и скажет, какой переменной не хватает —
чтобы забытый секрет нельзя было случайно оставить дефолтным.

Поднимутся три контейнера, БД проинициализируется, создастся администратор. Открывай:

- **Приложение:** http://localhost:8080
- **API + Swagger:** http://localhost:8000/docs
- Админ: имя и пароль из `ADMIN_USERNAME` / `ADMIN_PASSWORD` в `.env`

> `ADMIN_PASSWORD` действует только когда админа ещё нет. Если он уже создан, пароль
> меняется в самом портале: имя в правом верхнем углу → «Профиль».
> То же с паролем базы: `POSTGRES_PASSWORD` применяется при первом создании тома с данными.

Остановить: `docker compose down` (данные сохранятся). Полный сброс: `docker compose down -v`.
Пересобрать после изменений в коде: `docker compose up -d --build`.

> Порты можно поменять в `docker-compose.yml`. Если меняешь порт фронтенда — поправь и
> `FRONTEND_ORIGIN` у бэкенда; если порт бэкенда — build-args `VITE_API_URL` / `VITE_WS_URL` у фронтенда.

Ниже — **ручной запуск без Docker** (для разработки с hot-reload).

## Запуск (по шагам)

### 1. База данных (PostgreSQL)

Из **корня проекта** (там, где лежит `docker-compose.yml`):
```bash
docker compose up -d
```
Поднимется контейнер `dnd_postgres` с базой `dnd_portal`. Логин и пароль берутся из `.env`
(`POSTGRES_USER` / `POSTGRES_PASSWORD`), порт — `127.0.0.1:5432`, то есть только с твоего компьютера.
Те же логин и пароль пропиши в `DATABASE_URL` в `backend/.env`.
Остановить: `docker compose down` (данные сохранятся). Полностью удалить с данными: `docker compose down -v`.

### 2. Бэкенд (FastAPI)

```bash
cd backend

# Окружение dnd-portal-out уже создано. Если его нет — создай один раз:
#   conda create -n dnd-portal-out python=3.11 -y

conda activate dnd-portal-out        # активировать окружение
pip install -r requirements.txt      # установить зависимости (один раз)

copy .env.example .env               # создать настройки (Windows).  В bash: cp .env.example .env

alembic upgrade head                 # создать/обновить таблицы в базе
python seed_admin.py                 # создать администратора (один раз)

uvicorn app.main:app --reload        # запустить сервер
```

- API работает на **http://127.0.0.1:8000**
- Интерактивная документация (можно тыкать эндпоинты): **http://127.0.0.1:8000/docs**
- Данные администратора берутся из `.env`: `ADMIN_EMAIL` и `ADMIN_PASSWORD` **обязательны**
  (без них `seed_admin.py` не создаст админа и скажет об этом), `ADMIN_USERNAME` — по умолчанию `admin`.

> 🔑 **Смени пароль после первого входа:** нажми на своё имя в правом верхнем углу →
> страница «Профиль» → «Смена пароля». Если пароль забыт, почтового восстановления в
> портале нет: администратор задаёт временный пароль в Админ-панели (кнопка 🔑 в строке
> пользователя) и передаёт его лично, после чего человек меняет его в своём профиле.

> 🔒 **Секреты обязательны.** `DATABASE_URL` и `JWT_SECRET` в `backend/.env` (или в
> переменных окружения) должны быть заданы — без них бэкенд не запустится и скажет, чего
> не хватает. Значений по умолчанию у них специально нет: приложение не должно работать
> с предсказуемым ключом из кода. `JWT_SECRET` сгенерируй командой
> `python -c "import secrets; print(secrets.token_urlsafe(48))"`; после его смены все
> текущие входы аннулируются — нужно войти заново.
> Данные первого админа — тоже из `.env`: `ADMIN_EMAIL` и `ADMIN_PASSWORD` обязательны
> (без них контейнер не стартует / `seed_admin.py` не создаст админа), `ADMIN_USERNAME`
> по умолчанию `admin`. Пароль смени сразу после первого входа (страница «Профиль»).

### 3. Фронтенд (React)

В **новом окне терминала**:
```bash
cd frontend
npm install                          # установить зависимости (один раз)
copy .env.example .env               # настройки (Windows).  В bash: cp .env.example .env
npm run dev                          # запустить
```
Открой в браузере **http://localhost:5173**.

## Как пользоваться

1. **Админ** входит (логин `admin`, пароль — из `ADMIN_PASSWORD` в `.env`) → вкладка **Админ-панель → Коды-подписки** →
   создаёт код на N дней. Код появляется в таблице.
2. **Игрок/мастер** регистрируется → вводит код на странице **Подписка** → получает доступ.
3. **Мастер** на странице «Комнаты» создаёт комнату (автоматически становится DM),
   получает **код-приглашение**, в комнате загружает карту и ставит на неё фишки персонажей и NPC.
   Карт можно загрузить несколько и переключаться между ними кнопкой ▶ в панели мастера —
   карта меняется у всех игроков сразу.
4. **Игроки** вводят код-приглашение → попадают в комнату.
5. Все **перетаскивают свои фишки мышью** — перемещения видны всем мгновенно (WebSocket).
   Мастер может двигать любые фишки.
   **Клик по фишке** открывает свиток героя, из которого она выставлена (у NPC и чужих героев —
   краткую карточку с жизнями и эффектами). Свой свиток открывается кнопкой «📜 Мой свиток»
   в шапке комнаты: он показывается поверх игры, поэтому связь с комнатой не рвётся.
6. Любой бросает **кости** (кнопки d4–d100 или формула `2d6+3`, преимущество/помеха для d20) —
   результат появляется в **общем журнале** у всех.
7. Для боя мастер жмёт **«Начать бой»** в панели «⚔ Порядок ходов»: портал бросает d20
   инициативы всем фишкам и выстраивает очередь. Значения можно поправить руками (модификатор
   ловкости портал не знает). Чей ход — видно и в списке, и золотым кольцом на карте.
   Свой ход игрок завершает сам кнопкой «Завершить свой ход».

### Сетка и линейка

Мастер задаёт в своей панели **размер клетки** в пикселях карты — так, чтобы линии легли
по клеткам картинки. Одна клетка — 5 футов. Сетка хранится у комнаты, поэтому появляется
сразу у всех; 0 выключает её.

Кнопка **📏 Линейка** над картой меряет расстояние: клик — начало, клик — конец, подпись
показывает футы и клетки. Пока линейка включена, карта не таскается и фишки не двигаются,
чтобы замер не сбился. Второй клик фиксирует замер, третий начинает новый, выключение
кнопки убирает линию. Линейка личная: другие игроки чужих замеров не видят.

### Скрытые броски мастера

У мастера под кнопками костей есть галочка **🔒 Скрытый бросок**. Такой бросок не приходит
игрокам ни в журнал, ни по WebSocket — они не узнают даже о самом факте броска. Нужен для
проверок, о которых партия знать не должна: заметил ли кто засаду, сработала ли скрытая
ловушка. У мастера запись помечена золотой пунктирной рамкой, чтобы не спутать с общей.

Игрок скрытый бросок сделать не может — сервер вернёт 403.

### Заметки и раздаточные материалы

Панель **📖 Материалы** — место для описаний, текстов писем, портретов NPC и карт городов.
Мастер создаёт материал заранее (заголовок + текст, можно прикрепить картинку), и он
остаётся **скрытым**, пока мастер не нажмёт 🙈. После этого материал появляется у партии
сразу, без перезагрузки. Кнопка 👁 убирает его обратно.

Пока материал скрыт, сервер не отдаёт его игрокам вообще — как и скрытые фишки.

### Слои фишек

Если фишки налезли друг на друга и нижнюю не подцепить мышью, откройте её через ⚙ в списке
фишек и нажмите **⬆ наверх** или **⬇ вниз**. Свою фишку так же может поднять и сам игрок.

### Засады: скрытые фишки

Мастер может поставить фишку **скрытой** (галочка при создании или кнопка 👁 в списке фишек).
Такая фишка не видна игрокам ни на карте, ни в порядке ходов — сервер вообще не отправляет
им эти данные, поэтому подсмотреть их в браузере нельзя. Мастер видит её полупрозрачной
с подписью «СКРЫТ». Когда засада сработала — нажмите 🙈, и фишка появится у всех сразу.
Если ход дошёл до скрытого, игроки видят: «Сейчас ходит кто-то, кого вы пока не видите».

### Два «персонажа»: свиток и фишка

Чтобы не путаться (в коде это `HeroSheet` и `Token`):

| | **Свиток героя** (`HeroSheet`) | **Фишка** (`Token`) |
|---|---|---|
| Что это | лист персонажа: характеристики, инвентарь, история | кружок на карте конкретной комнаты |
| Где живёт | страница «Свиток героя», у игрока их может быть много | внутри одной комнаты |
| Зачем | описать, кто твой герой | показать, где он стоит и сколько у него HP сейчас |

Один свиток можно выставить фишками сразу в нескольких комнатах (кнопка «На карту»).
Свиток — источник правды: меняешь в нём имя, портрет или максимум HP — это подтягивается
во все его фишки. Текущее боевое HP живёт у фишки и в свиток не возвращается.
Фишка может существовать и без свитка — например, NPC, которого мастер создал прямо на карте.

### Роли
- **player** — играет: двигает свою фишку, бросает кости.
- **dm** (мастер) — всё выше + панель мастера (карта, фишки, участники). Игрок становится DM,
  создав комнату.
- **admin** — админ-панель: пользователи (роль, блокировка, подписка), коды, комнаты, фишки.

## Проверка «в двух окнах» (реалтайм)

1. Открой комнату в обычном окне браузера (мастер) и в окне «инкогнито» (игрок, вошедший под
   другим аккаунтом и присоединившийся по коду).
2. Подвинь фишку в одном окне — она сдвинется во втором.
3. Брось `2d6+3` — запись появится в журнале у обоих.

## Частые проблемы

- **CORS-ошибка в браузере** — проверь, что `FRONTEND_ORIGIN` в `backend/.env` совпадает с адресом
  фронтенда (`http://localhost:5173`).
- **Фронтенд не видит API** — проверь `frontend/.env` (`VITE_API_URL`, `VITE_WS_URL`) и что бэкенд запущен.
- **Не подключается к БД** — убедись, что контейнер поднят (`docker compose ps`) и порт 5432 свободен.
- **`npm run build` ругается на типы** — для разработки достаточно `npm run dev` (он не проверяет типы строго).

## Миграции базы данных (Alembic)

Схемой базы заведует **Alembic**. Приложение само таблицы больше не создаёт — так у всех
(и у тебя, и у того, кто развернёт проект с нуля) база получается одинаковой, а каждое
изменение схемы остаётся в истории в `backend/alembic/versions/`.

В Docker миграции применяются сами при старте контейнера. Вручную — из папки `backend`:

```bash
alembic upgrade head      # применить все миграции (создать/обновить таблицы)
alembic current           # какая ревизия сейчас в базе
alembic history           # вся история изменений схемы
```

**Если поменял `models.py`** (добавил колонку, таблицу и т.п.):

```bash
alembic revision --autogenerate -m "что изменил"   # создать файл миграции
# ОБЯЗАТЕЛЬНО открой новый файл в alembic/versions/ и проверь, что там написано
alembic upgrade head                               # применить
```

> ⚠ Никогда не правь таблицы руками через SQL — иначе база и миграции разъедутся.
> Проверить, что база совпадает с моделями: `alembic check` (ответ
> «No new upgrade operations detected» = всё сходится).

Откатить последнюю миграцию: `alembic downgrade -1`.

## Про архитектуру (микросервисы?)

Проект — **контейнеризированный модульный монолит**, а не микросервисы. Бэкенд — один процесс
FastAPI с роутерами по доменам (`auth`, `codes`, `rooms`, `tokens`, `dice`, `heroes`, `admin`, `ws`)
и одной общей БД. В Docker он разбит на 3 сервиса (frontend/backend/db) — это разделение по слоям,
а не по бизнес-доменам.

Для игры с друзьями это оптимально. Полноценные микросервисы потребовали бы: отдельных сервисов
по bounded-context'ам, БД на сервис, API-gateway, замены in-memory `ws_manager` на Redis pub/sub
и шины сообщений — избыточно на таком масштабе. Модульность роутеров даёт удобные «швы», если
захочется пилить на сервисы позже.
